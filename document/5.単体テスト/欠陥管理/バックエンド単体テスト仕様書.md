# Defect（欠陥）バックエンド単体テスト仕様書  
対象：Rust / DefectCommand（CRUD 基本動作）

本仕様書は、Defect（欠陥）に関するバックエンド処理の  
**CRUD の基本動作** を確認するための単体テスト項目を定義する。

バリデーションは未実装のため、  
**キー情報の有無** と **項目の有無** を中心に確認する。

---

# 1. 対象範囲（Scope）

- DefectCommand（create / update / delete / list）
- JSON DB（mock_db.json）
- ID 採番処理
- キー情報による検索・更新・削除の制御

---

# 2. テスト観点

| 観点           | 説明                                |
| -------------- | ----------------------------------- |
| 検索（list）   | キーの有無で取得件数が変わる        |
| 作成（create） | キーの有無で登録可否が変わる        |
| 更新（update） | キーの有無で更新可否が変わる        |
| 削除（delete） | キーの有無で削除可否が変わる        |
| ID 採番        | create 時に ID が自動採番される     |
| 欠番採番       | 欠番があっても最大値+1 で採番される |
| 存在しない ID  | update/delete 時に NotFound を返す  |

---

# 3. テストケース一覧

## 3.1 list_defect（検索）

### ✔ LS-D01-01：キー情報なし → 取得しない  
- 入力：key=""  
- 期待：Err

### ✔ LS-D01-02：単独キー → 該当データのみ取得  
- 入力：project_id="P1"  
- 期待：P1 のデータのみ返る

---

## 3.2 create_defect（作成）

### ✔ CR-D01-01：キー情報なし → 登録しない  
- 入力：project_id=""  
- 期待：Err、DB 変更なし

### ✔ CR-D01-02：キー情報のみ → 初期値で登録  
- 入力：project_id="P1"  
- 期待：DEFECT-1 が採番され、初期値で登録

### ✔ CR-D01-03：全情報あり → 全項目を登録  
- 入力：全項目  
- 期待：内容一致で登録

### ✔ CR-D01-04：欠番がある場合でも正しく採番される  
- 前提：["DEFECT-1", "DEFECT-3"]  
- 入力：project_id="P1"  
- 期待：新 ID = "DEFECT-4"

---

## 3.3 update_defect（更新）

### ✔ UP-D01-01：キー情報なし → 更新しない  
- 入力：id=""  
- 期待：Err

### ✔ UP-D01-02：キー情報のみ → 更新しない  
- 入力：id="D1", project_id="P1"  
- 期待：内容変更なし

### ✔ UP-D01-03：全情報あり → 全項目を更新  
- 入力：全項目  
- 期待：内容一致で更新

### ✔ UP-D01-04：存在しない ID → 更新不可  
- 入力：id="D999"  
- 期待：Err("Not found")

---

## 3.4 delete_defect（削除）

### ✔ DL-D01-01：キー情報なし → 削除しない  
- 入力：id=""  
- 期待：Err

### ✔ DL-D01-02：キー情報あり → 削除成功  
- 入力：id="DEFECT-1"  
- 期待：該当 ID が DB から消える

### ✔ DL-D01-03：存在しない ID → 削除不可  
- 入力：id="D999"  
- 期待：Err("Not found")

---

# 4. 補足

- create/update/delete は DefectCommand::xxx_impl を直接テストする。
- DB は mock_db.json を使用し、本番 DB には影響しない。
