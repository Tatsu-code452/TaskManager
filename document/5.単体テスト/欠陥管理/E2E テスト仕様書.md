# Defect（欠陥）E2E テスト仕様書  
対象：React（UI）＋ Tauri（Rust バックエンド API）＋ JSON DB

本仕様書は、欠陥（Defect）に関する  
**UI → Tauri → Rust → JSON DB の結合動作**を確認するための  
E2E（End-to-End）テスト項目を定義する。

---

# 1. 対象範囲（Scope）

- 欠陥一覧画面（Defect List）
- 欠陥新規作成画面（Defect New）
- 欠陥編集画面（Defect Edit）
- Tauri invoke（create / update / delete / list）
- JSON DB（mock_db.json）
- ID 採番処理
- タスク紐付け（あり／なし）

---

# 2. テスト観点

| 観点 | 説明 |
|------|------|
| 一覧表示 | 欠陥一覧が正しく表示される |
| 新規作成 | UI 入力 → invoke → Rust → DB → UI 反映 |
| 編集 | ステータス・内容の更新 |
| 削除 | UI 操作で欠陥が削除される |
| バリデーション | 必須項目エラーの表示 |
| NotFound | 存在しない ID の編集画面遷移 |
| ID 採番 | 欠番がある場合でも最大値+1 で採番される |
| タスク紐付け | 紐付けあり／なしの両方を確認 |

---

# 3. テストケース一覧

## 3.1 DF-01：欠陥一覧が表示される

- 前提：DB に既存データあり  
- 操作：`/project/P1/defects` を開く  
- 期待：  
  - 「新規欠陥」ボタンが表示される  
  - 既存の欠陥（例：入力チェック漏れ）が表示される  

---

## 3.2 DF-02：新規欠陥作成（タスク紐付けなし）

- 操作：  
  1. 「新規欠陥」クリック  
  2. タイトル・内容・重大度・ステータスを入力  
  3. タスク紐付け＝なし  
  4. 保存  
- 期待：  
  - invoke("create_defect") が呼ばれる  
  - 一覧に新規欠陥が追加される  
  - タスク欄は「-」になる  

---

## 3.3 DF-03：新規欠陥作成（タスク紐付けあり）

- 操作：  
  1. 「新規欠陥」クリック  
  2. タスク紐付け＝T1 を選択  
  3. 保存  
- 期待：  
  - invoke payload に task_id="T1" が含まれる  
  - 一覧に T1 が表示される  

---

## 3.4 DF-04：欠陥編集（ステータス変更）

- 操作：  
  1. 欠陥一覧の「✎」をクリック  
  2. ステータスを Open → Fixed に変更  
  3. 保存  
- 期待：  
  - invoke("update_defect") が呼ばれる  
  - 一覧のステータスが Fixed に更新される  

---

## 3.5 DF-05：欠陥削除

- 操作：  
  1. 欠陥一覧の「削除」クリック  
  2. 確認ダイアログで OK  
- 期待：  
  - invoke("delete_defect") が呼ばれる  
  - 一覧から該当欠陥が消える  

---

## 3.6 DF-06：必須項目エラー（タイトルなし）

- 操作：  
  1. 「新規欠陥」クリック  
  2. タイトル空のまま保存  
- 期待：  
  - invoke は呼ばれない  
  - UI に「タイトルは必須です」表示  

---

## 3.7 DF-07：存在しない ID を直接 URL で開いた場合

- 操作：  
  `/project/P1/defects/DEFECT-999/edit` に直接アクセス  
- 期待：  
  - invoke("get_defect") が Err を返す  
  - UI に「データが見つかりません」表示  

---

## 3.8 DF-08：ID 採番確認（欠番がある場合）

- 前提：DB に DEFECT-1, DEFECT-3 が存在  
- 操作：  
  1. 新規欠陥作成  
  2. 保存  
- 期待：  
  - invoke("create_defect") が呼ばれる  
  - Rust 側で DEFECT-4 が採番  
  - UI に DEFECT-4 が表示される  

---

# 4. 補足

- Playwright を使用して UI を自動操作する  
- Tauri dev server（http://localhost:1420）を対象とする  
- JSON DB はテスト用 mock_db.json を使用  
- CRUD ロジックは Rust 側の単体テストで保証済み  
- 本仕様書では **画面操作フローの正しさ** を検証する
