# 📘 Tag 設定要領（Domain / Area / Topic）

Issue / Defect に付与するタグは、  
**Domain（技術領域） / Area（業務領域） / Topic（テーマ）**  
の 3 種類に分類する。

タグは任意入力だが、  
分類軸を明確にすることで **フィルタしやすく、意味のあるデータ管理が可能** となる。

---

## 1. Tag の目的

- 課題・欠陥の分類を明確にする  
- フィルタ・検索性を高める  
- 技術領域・業務領域・テーマを横断的に整理する  
- プロジェクトの規模が大きくなっても破綻しないタグ体系を維持する  

---

# 2. Domain（技術領域）

**「どの技術スタック・どの層に関係する問題か」** を示す分類。

### 主な例

| 値 | 説明 |
|----|------|
| Frontend | React / TypeScript / UI 全般 |
| Backend | Rust / API / ビジネスロジック |
| API | REST / JSON / エンドポイント設計 |
| DB | SQL / Schema / Migration |
| UI | レイアウト / スタイル / コンポーネント |
| UX | 操作性 / 導線 / 使い勝手 |
| Infra | Tauri / OS / FileSystem |
| Build | ビルド / パッケージ / CI |

### 使いどころ
- React の問題か Rust の問題かを明確にしたい  
- 技術者ごとに担当領域を分けたい  
- 技術領域別にフィルタしたい  

---

# 3. Area（業務領域）

**「どの工程に関係する課題か」** を示す分類。

### 主な例

| 値 | 説明 |
|----|------|
| 仕様 | 要件定義・仕様の不備 |
| 設計 | データ構造・API 設計・UI 設計 |
| 実装 | コーディング作業 |
| テスト | バグ検証・テストケース |
| 運用 | リリース後の改善・運用課題 |
| ドキュメント | README・仕様書・コメント |

### 使いどころ
- 「これは仕様の問題？実装の問題？」を明確にしたい  
- 工程別に課題を整理したい  
- レビュー時に見落としを防ぎたい  

---

# 4. Topic（テーマ）

**「具体的に何に関する問題か」** を示す分類。  
最も細かい粒度で、検索性に最も効く。

### 主な例

| 値 | 説明 |
|----|------|
| 採番 | ID の生成・重複防止 |
| バリデーション | 入力チェック・制約 |
| 認証 | ログイン・セッション |
| 権限 | ロール・アクセス制御 |
| パフォーマンス | 遅延・最適化 |
| UI レイアウト | 配置・崩れ |
| スタイル | CSS / 見た目 |
| データ構造 | モデル・型定義 |
| API 連携 | 通信・レスポンス |
| 保存処理 | JSON / DB 書き込み |
| 日付処理 | タイムゾーン・フォーマット |

### 使いどころ
- 課題の本質を一言で表したい  
- 同じテーマの課題を横断的に見たい  
- 後から検索しやすくしたい  

---

# 5. タグ付けルール

### ✔ Domain = 技術領域  
例：Frontend / Backend / DB

### ✔ Area = 工程  
例：仕様 / 設計 / 実装

### ✔ Topic = 具体的テーマ  
例：採番 / バリデーション / スタイル

### ✔ 1 つの Issue / Defect に複数タグを付けてよい  
例：

```json
[
  { "tag_type": "Domain", "value": "Backend" },
  { "tag_type": "Area", "value": "仕様" },
  { "tag_type": "Topic", "value": "採番" }
]
```

### ✔ 迷ったら Topic を優先  
Topic が最も検索性に効く。

# 7. まとめ

- **Domain** → 技術領域  
- **Area** → 開発工程  
- **Topic** → 具体的テーマ  
- 3 軸でタグを付けることで、  
  - フィルタしやすい  
  - 意味が明確  
  - データが育つ  
  - UI での検索性が高い
